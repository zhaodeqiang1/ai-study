# 测试契约

每次非平凡代码开发必须满足本契约。

## 基本要求

- 新增业务逻辑必须优先新增单元测试。
- 修改已有业务逻辑必须优先新增或更新单元测试。
- 修复 Bug 必须优先新增回归测试。
- 涉及 SQL、Mapper、权限、状态流转、AI 模型调用等高风险逻辑时，必须说明测试覆盖策略。

## 当前项目现状

当前仓库尚未发现稳定的 `src/test` 测试目录，也未发现明确的 JUnit、Mockito 或 Spring Boot Test 依赖。

因此，在真正新增第一批单元测试前，需要先确认测试框架和依赖策略。

## 无法新增测试时

如果本次变更无法新增单元测试，必须在 AI 开发记录中说明：

- 为什么无法新增。
- 替代验证方式。
- 剩余风险。
- 后续补测建议。

不能只写“无需测试”。

## 最低验证

即使无法新增单元测试，也必须尽量执行最小有效验证：

- 后端模块编译。
- 后端模块测试命令。
- 前端构建。
- SQL 静态检查。
- 关键逻辑人工审查。

## 审查门要求

审查门必须检查：

- 是否应该新增或更新单元测试。
- 是否已有测试覆盖核心分支。
- 未补测试的原因是否合理。
- 替代验证是否足够。
