# 代码审查门检查清单

非平凡代码实现后，必须经过审查门，再进入验证和收尾。

## 适用范围

满足任一条件时必须执行审查门：

- 新增功能。
- 修改业务逻辑。
- 修改权限、菜单、角色、数据范围。
- 修改 SQL、Mapper、事务或状态流转。
- 修改 Spring AI、模型调用、Prompt、RAG、向量检索或工具调用。
- 修改前端页面、按钮权限或 API 调用。
- 修复生产、安全、数据一致性相关问题。

## 必读文件

- `harness-engineering/workflows/review-workflow.md`
- `harness-engineering/skills/code-review.md`
- `harness-engineering/constraints/security-constraints.md`
- 本次变更涉及的文件或 Git diff。

## 审查内容

- 是否满足需求验收标准。
- 是否只修改了允许范围。
- 是否存在权限绕过。
- 是否存在敏感信息泄露。
- 是否有参数校验。
- 是否有错误处理。
- 是否有日志和审计。
- 是否有事务和并发风险。
- 是否有 SQL 回滚和验证。
- 是否有可执行验证命令。
- 是否应该新增或更新单元测试。
- 未新增测试时是否说明原因、替代验证和剩余风险。
- 是否满足适用契约。
- 是否需要评测集以及评测结果。
- 是否有未记录的风险或后续事项。

## 审查结果

审查后必须明确：

- `Pass`：可以进入验证。
- `NeedsFix`：必须先修复问题，再重新审查。
- `Blocked`：需要用户确认或外部条件。

## 记录要求

审查结果必须写入对应 AI 开发记录：

- 审查方式。
- 发现问题。
- 修复情况。
- 是否通过。
