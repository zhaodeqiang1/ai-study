# 审查流程

代码审查流程。

审查既可以是用户主动发起的独立任务，也可以是新功能或 Bug 修复后的主线审查门。

## 1. 确定审查对象

优先读取用户指定文件或 Git diff。

如果是实现后的审查门，审查对象是本次 AI 修改过的文件和对应需求验收标准。

## 2. 审查风险

重点检查：

- 安全。
- 权限。
- 数据一致性。
- 事务。
- 并发。
- 错误处理。
- 日志。
- 测试，尤其是是否满足 `harness-engineering/contracts/testing-contract.md` 和 `harness-engineering/checklists/unit-test-checklist.md`。
- 适用契约是否满足。
- 是否需要评测集以及评测结果是否记录。

## 3. 输出问题

问题优先，摘要靠后。

每条问题说明：

- 文件。
- 位置。
- 影响。
- 修复建议。

## 4. 审查结论

审查门必须给出结论：

- `Pass`：没有阻塞问题，可以进入验证。
- `NeedsFix`：发现必须修复的问题，修复后重新审查。
- `Blocked`：需要用户确认或外部条件。

## 5. 不自动修改

除非用户要求修复，否则审查任务只输出发现。

实现后的审查门如果发现问题，允许在同一任务中进行最小修复，但修复后必须重新执行审查门。

## 6. 反馈沉淀

如果审查发现重复风险、生产安全风险、验证缺失或文档缺失，应建议更新对应 Harness 文件。

如果审查发现契约缺失或评测缺失，应建议更新 `harness-engineering/contracts` 或 `harness-engineering/evaluations`。

如果审查发现应补单元测试但未补，且缺少合理原因、替代验证、剩余风险和后续补测建议，应判定为 `NeedsFix`。
