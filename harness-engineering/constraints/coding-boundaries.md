# 编码边界

## 最小变更原则

AI 每次只做用户要求的变更，不做顺手重构。

## 文件访问边界

优先修改任务归属模块下的文件。

除非用户明确要求，不修改：

- 认证服务。
- 网关服务。
- 公共模块。
- 父级 Maven 配置。
- 全局安全配置。
- 全局前端路由守卫。
- Docker 和部署脚本。

## 代码风格

必须遵循 RuoYi Cloud 现有风格：

- Controller 保持薄层。
- Service 承载业务逻辑。
- Mapper 只做数据访问。
- XML 放在对应 `resources/mapper` 下。
- 前端 API 独立放在 `src/api`。
- 前端页面独立放在 `src/views`。
- 权限标识、菜单、按钮权限要与 RuoYi 规范一致。

## 依赖约束

不随意新增第三方依赖。

确需新增依赖时，必须说明：

- 为什么现有能力无法满足。
- 依赖放在哪个模块。
- 是否影响父级构建。
- 是否有生产风险。

## 生成代码约束

AI 生成代码时必须：

- 使用明确的包名和模块边界。
- 不硬编码密钥、Token、模型 Key、数据库连接。
- 不吞异常。
- 不输出敏感日志。
- 不引入不可控的自动执行逻辑。
