# 需求设计技能

本技能用于把用户想法整理成后续开发可直接使用的需求文档。

## 使用时机

当用户提出以下任务时使用：

- 需求设计。
- 业务方案设计。
- PRD 或功能说明。
- 影响范围评估。
- 需求拆分。
- 创建或更新需求文档。
- 开发前先明确验收标准和边界。

## 固定输出位置

需求文档必须写入：

```text
docs/requirements/REQ-YYYYMMDD-NNN-<english-slug>.md
```

并登记到：

```text
docs/requirements/INDEX.md
```

文件名使用英文，正文使用中文。

## 需求编号规则

- 格式：`REQ-YYYYMMDD-NNN`。
- `YYYYMMDD` 使用当前日期。
- `NNN` 从当天已有最大编号继续递增。
- 如果是修改已有需求，优先更新原需求文件，不新建重复需求。

## 需求设计流程

1. 读取 `docs/requirements/README.md`、`docs/requirements/REQ-000-template.md` 和 `harness-engineering/contracts/requirement-contract.md`。
2. 判断用户输入是否足以形成需求。
3. 如果缺少关键信息，先写入“待确认问题”，不要臆造业务规则。
4. 明确背景、目标、范围、非目标、业务规则和验收标准。
5. 按 `harness-engineering/checklists/impact-scope-checklist.md` 评估影响范围。
6. 明确后续应进入哪个流程：功能开发、Bug 修复、审查、SQL 迁移、AI 功能评测或反馈循环。
7. 写入需求文档并更新需求索引。

## 影响范围评估

每个非平凡需求必须说明：

- 后端模块影响。
- 前端页面和 API 影响。
- 数据库、SQL、Mapper、索引和数据迁移影响。
- 权限、菜单、角色、数据权限影响。
- 配置、部署、环境变量和第三方依赖影响。
- Spring AI Alibaba、模型调用、Prompt、RAG、工具调用和 Token 统计影响。
- 测试影响，包括单元测试、集成验证、前端构建和 SQL 验证。
- 生产风险、回滚方式、灰度或部署顺序。
- 不改动范围。

如果某项不适用，写明“不涉及”，不要省略。

## 验收标准要求

验收标准必须可判断。优先使用：

- 页面、接口、权限、SQL、日志、配置、测试命令等可验证条目。
- 具体状态流转和边界条件。
- 明确“不通过”的场景。

禁止只写：

- “功能正常”。
- “体验好”。
- “满足业务需要”。

## 状态建议

- 信息不足：`Draft`。
- 已完成需求设计但未获实现确认：`Reviewed`。
- 用户明确确认可进入实现：`Approved`。
- 已实现：`Implemented`。
- 已验证：`Verified`。

## 后续流程衔接

需求文档末尾必须写清：

- 推荐后续流程。
- 推荐读取的技能或契约。
- 需要先确认的问题。
- 是否需要评测集。

后续开发必须引用该需求编号，并在 AI 开发记录中关联需求文档。
