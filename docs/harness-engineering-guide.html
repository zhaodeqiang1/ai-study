Harness Engineering 说明文档

本文档说明当前项目中 Harness Engineering 的目录结构、文件职责、治理机制、需求追溯、AI 开发记录、SQL 回滚约束和反馈循环流程。它是给人和 AI 共同阅读的工程说明，不参与运行时业务逻辑。

1. 总览

本项目使用 AGENTS.md 作为 Codex 默认入口，通过 harness-engineering 目录承载 AI 开发约束。核心目标是让 AI 在写代码前先明确边界、需求、上下文、验证方式和记录要求。

1 Codex 默认入口
15 Harness 子目录
12 需求体系文件
3 治理闭环文件
最高优先级规则：AI 不得默认扫描全仓库，不得修改认证、网关、公共模块、父级构建配置，不得写入真实密钥，不得声称执行过未执行的测试或构建。

2. 为什么这样设计

Harness Engineering 的设计目标不是“多写文档”，而是把 AI 开发中最容易失控的部分前置成规则：读什么、不能改什么、为什么改、怎么验证、怎么回滚、怎么复盘。这样可以把一次次临时对话变成可追溯、可复用、可治理的工程资产。

问题和设计映射

原始问题 设计方案 解决效果 代价
AI 容易一上来扫描大量文件，浪费 Token，也更容易被无关代码带偏。 AGENTS.md + loaders/progressive-loading.md + token-budget.md 先读入口和项目地图，再按任务分支读取局部上下文，减少无效读取。 用户或 AI 需要先判断任务类型，不能直接无脑全仓库搜索。
AI 可能修改认证、网关、公共模块、父级构建配置等高风险区域。 AGENTS.md 绝对禁止事项 + constraints/coding-boundaries.md + file-access-policy.md 把高风险区域设为默认禁区，只有用户明确授权才扩大修改范围。 跨模块改造前需要更多说明和确认。
后续只能看到“AI 改了什么”，看不到“为什么要改”。 docs/requirements 需求编号、需求索引、状态流转、验收标准。 每个非平凡功能都有需求来源和验收标准，方便追溯业务动机。 非平凡变更前需要补需求记录。
总变更日志粒度太粗，无法还原一次 AI 开发的背景、决策和风险。 harness-engineering/records 独立 AI 开发记录。 每次非平凡 AI 开发都有背景、改动、关键决策、验证、风险和后续事项。 需要维护记录索引和单次开发记录。
SQL 迁移只写正向脚本，生产回滚风险高。 sql-migration-template.md + sql-migration-checklist.md + database-mybatis.md 强制要求正向 SQL、回滚 SQL 或回滚方案、数据影响、部署顺序、验证 SQL、不可逆说明。 写 SQL 前需要明确部署和回滚策略。
验证停留在“应该测试”，没有明确执行命令。 skills/testing-verification.md 提供 Maven、模块级测试、前端构建命令；不能执行时必须说明原因。 本地缺少依赖时，需要明确记录未验证风险。
代码开发可以只跑构建，不补单元测试，导致业务逻辑缺少回归保护。 contracts/testing-contract.md + checklists/unit-test-checklist.md 要求非平凡代码开发优先新增或更新单元测试；无法补测试时必须说明原因、替代验证、剩余风险和后续补测建议。 当前项目尚未建立测试依赖，第一次补测试前需要确认测试框架和依赖策略。
需求设计如果只停留在对话里，后续开发只能知道“要做什么”，不知道影响哪些模块、为什么做、怎么验收。 .agents/skills/requirements-design + skills/requirements-design.md + checklists/impact-scope-checklist.md 把需求设计、影响范围评估、需求编号和固定文档落点合成一个前置技能，需求文档统一写入 docs/requirements。 开发前多一步需求沉淀，但后续实现、审查、测试和发布都有稳定输入。
新需求代码实现后如果直接进入验证，可能漏掉权限、安全、边界、事务、SQL 回滚等代码质量问题。 workflows/review-workflow.md + skills/code-review.md + checklists/code-review-gate-checklist.md 实现后先进入审查门，审查通过才进入验证；发现问题则回到实现修复，再重新审查。 开发流程多一个门禁步骤，但能显著降低生产缺陷和返工。
用户反馈容易只停留在对话里，下一次 AI 还会犯同类错误。 governance/feedback-loop.md + anti-entropy-governance.md 把反馈分类沉淀到需求、约束、技能、流程、记录或检查清单中。 每次重要反馈都要判断应该更新哪个 Harness 文件。
项目演进后，文档可能越来越多但互相冲突或过期。 governance/entropy-audit-checklist.md 定期检查需求、记录、SQL、验证、项目地图和 Harness 自身是否漂移。 需要周期性维护，尤其是新增模块或流程变化后。
规则写了很多，但没有明确“做到什么才算合格”。 harness-engineering/contracts 用需求契约、AI 变更契约、审查门契约、SQL 契约和 AI 功能契约定义合格标准。 每次非平凡变更需要声明适用契约。
契约如果没有评测，仍然只能靠主观判断。 harness-engineering/evaluations 用评测集、用例和评分标准检查 Harness 治理质量和 AI 功能质量。 需要维护评测用例，并在 AI 功能或治理变化时补充评测。

核心设计取舍

设计取舍 原因 结果
入口使用 AGENTS.md，不自定义额外入口。 贴合 Codex 默认读取机制，减少入口分裂。 AI 进项目先读一个权威入口，其他文件由入口渐进引导。
需求放在 docs/requirements，不放在 Harness 内部。 需求是项目级事实，不只是 AI 约束文件。 人、AI、后续维护者都能从项目文档中理解业务动机。
总变更日志和独立 AI 开发记录分离。 总日志适合概览，独立记录适合追溯一次完整过程。 既能快速看变化，也能深入看背景、决策、验证和风险。
技能按任务加载，不一次性全部读取。 不同任务需要的上下文不同，全部加载会浪费 Token。 后端、前端、SQL、Spring AI、Review 各走自己的最小上下文。
SQL 和验证规则写成硬约束。 这两类最容易影响生产安全，不能只写建议。 AI 生成 SQL 或完成开发时必须考虑回滚、验证和未验证风险。
代码实现和代码审查分成两个阶段。 实现阶段关注完成需求，审查阶段关注风险、边界和生产质量。 新功能、Bug 修复、SQL、AI 能力和前端变更都能在验证前先经过质量门禁。
契约和评测分离。 契约定义合格标准，评测集定义如何检查这些标准。 规则不只停留在文档里，而是能用案例和评分标准持续检查。
Codex 原生技能和 Harness 详细技能文档分离。 .agents/skills 负责 Codex 自动发现，harness-engineering/skills 负责详细规则治理。 技能符合 Codex 规范，同时避免重复维护两套长文档。

10. SQL 迁移和验证体系

SQL 变更是生产风险最高的区域之一，因此 Harness 将 SQL 规则从“考虑回滚”提升为“必须提供回滚 SQL 或明确回滚方案”。

文件 作用
sql/harness/migrations/README.md 定义 Harness / AI 业务 SQL 迁移文件命名和必填内容。
harness-engineering/templates/sql-migration-template.md SQL 迁移模板，要求正向 SQL、回滚 SQL、数据影响、部署顺序、验证 SQL、不可逆说明。
harness-engineering/checklists/sql-migration-checklist.md SQL 迁移检查清单，禁止只有正向 SQL，禁止无说明地执行不可逆 SQL。

11. 关键规则速查

场景 必须执行 沉淀位置
需求设计 需求必须形成编号和文档，固定写入 docs/requirements，并包含影响范围评估。 .agents/skills/requirements-design
非平凡功能 创建或确认需求编号、登记需求索引、定义验收标准。 docs/requirements
非平凡 AI 开发 新增独立 AI 开发记录，包含背景、改动、关键决策、验证、风险、后续事项。 harness-engineering/records
SQL/migration 提供正向 SQL、回滚 SQL 或回滚方案、数据影响、部署顺序、验证 SQL、不可逆说明。 sql/harness/migrations
用户反馈 判断是需求缺失、规则缺失、技能缺失、流程缺失、验证缺失还是记录缺失，并更新对应文件。 harness-engineering/governance
验证 能运行命令时必须运行，不能运行时必须说明原因并记录风险。 harness-engineering/skills/testing-verification.md
单元测试 非平凡代码开发必须优先新增或更新单元测试；无法补测试时必须说明原因和替代验证。 harness-engineering/contracts/testing-contract.md
契约 非平凡功能必须声明并检查适用契约。 harness-engineering/contracts
评测集 AI、SQL、审查门或 Harness 治理变更必须说明评测集结果或不适用原因。 harness-engineering/evaluations
